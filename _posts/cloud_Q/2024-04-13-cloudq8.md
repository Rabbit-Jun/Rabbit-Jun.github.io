2.  하이퍼바이저 기반 가상화는 하이퍼바이저라는 소프트웨어 계층을 통해 구현된다. (O/X)
3.  하이퍼바이저의 별칭은 "Supervisor of Supervisors"이다. (O/X)

5.  Type 2 하이퍼바이저는 호스트 운영체제를 통해 하드웨어 리소스에 액세스한다. (O/X)
6.  Type 1 하이퍼바이저는 클라우드 인프라, 고성능 컴퓨팅 등에 적합하다. (O/X)
7.  Type 2 하이퍼바이저는 개발, 테스트, 교육 목적 등 경량화된 사용에 적합하다. (O/X)
8.  Type 1 하이퍼바이저는 별도의 Host OS가 없기 때문에 오버헤드가 적다. (O/X)
9.  Type 2 하이퍼바이저는 하드웨어 수준의 가상화 지원으로 오버헤드가 줄어들었다. (O/X)

### 문제
1. 하이퍼바이저 기반 가상화: __인 자원과 __인 자원을 구분하며 __라고 불리는 가상화 층(virtualization S/W layer)을 통해 구현되고 “Supervisor of Supervisors” 라고도 함
실제 __ 자원의 복잡함을 숨기고 단일하며 일관된 __ 자원들을 제공

2. 하이퍼 바이저는 __ 위치에 따라 두가지로 분류하며 __은 ‘Native’ (or ‘Bare Metal’)형 하이퍼바이저
물리 서버 위에 __ 없이 직접 설치되고, (직접적인/간접적인) 하드웨어에 접근하여__ 자원을 가상화하며 __는 ‘Hosted’ 하이퍼바이저 운영체제에 설치(하나의 프로그램으로)되어 실행되며 __ 운영체제를 통해 __ 리소스에 액세스
3. Type 1의 장점으로는 별도의 __ OS가 없기 때문에 __가 적다(메모리 사용량, 가상화에 따른 부하 낮음)
4. Type 1의 단점으로는 지원하는 장치가 __기반 가상화에 비해 적고 __ 이슈가 있을 수 있으며(많이 개선됨) 관리와 유지보수를 위한 컴퓨터나 __이 필요할 수 있음
5. Type 2의 장점으로는 가상의 하드웨어를 __하기 때문에 __ 운영체제에 크게 제약상항이 없으며(Host OS가 이미 지원하는 장치를 모두 활용 가능) __ 위에서 소프트웨어로 구동되기 때문에 관리 및 __가 용이함
6. Type 2의 단점으로는 OS위에 __가 얹히는 방식이이 때문에 __가 클 수 있으며 Host OS의 가용성에 따라 하이퍼바이저와 __ OS가 영향을 받는다
7. __ 는 가상화된 환경에서 작동하며, __ 자원에 대한 직접적인 접근이 필요함 다시 말하면, __가 실제 물리 자원을 사용하는 것과 유사한 방식의 __를 통해 GuestOS에 자원을 제공해야 함
8. 하이퍼바이저는 여러 GuestOS에 __ 자원의 직접적인 액세스를 허용하는 동시에, 이들 간의 __ 분배와 조정을 위한 효율적인 __가 가능해야 함
9. 하이퍼바이저의 두가지 설계 방법으로는 전가상화(Full Virtualization)와 __가 있다.
10. 전가상화 : Guest OS의 수정이 필요 없도록 __ 전체를 가상화 하는 기술로 Guest OS는 자신이 가상머신의 __인지 인지하지 못하며 __ 입장에서는 가상하된 하드웨어를 __ 하드웨어로 인지하며 하드웨어를 완전히 가상화 하기 때문에 Guest OS의 커널 등의 수정이 필요 (함/없음)
11. __ 명령어:시스템의 설정이나 구성을 __할 수 있는 명령어로 하드웨어 자원에 (직접적/간접적)으로 접근하거나 운영 체제의 중요한 설정을 변경 Ex) 메모리 관리, 입출력 장치의 제어 등
12. __ 명령어: 시스템의 설정이나 구성에 영향을 주지 않는 명령어들
13. CPU 보호 링 (CPU Protection Ring): 운영 체제의 보안과 __을 유지하기 위해 설계된 메커니즘으로 CPU가 제공하는 다양한 __ 수준을 나타내며 0에서 3까지의 __ 개의 링으로 구분됨 (낮은 수자일수록 더 높은 권한)
14. __: 컴퓨터 시스템에서 하드웨어와 소프트웨어가 최고 권한으로 동작하는 __모드(특권모드)와 일반 사용자 레벨에서 동작하는 __모드(비특권모드) 사이를 전환할 수 있도록 하는 __ 운영 모드
15. 비특권모드의 사용자 프로세스가 시스템 리소스 엑세스가 필요하다면 __을 사용하면 된다.
16. 시스템콜(System Call)의 종류로는 파일 시스템, __ 관리, 메모리 관리, __, 입출력, __ 정보 이 있다
17. __: 소프트웨어 인터럽트의 한 형태로 운영 체제에게 제어를 넘겨주는 역할을 수행하며 프로세스의 __ 모드가 사용자 모드(user mode)에서 커널 모드(kernel mode)로 전환됨
18. 가상화 환경에서 __가 가상 머신이 수행하는 명령어를 제어하기 위한 방법으로 트랩이 사용됨
19. __: 가상 머신이 하드웨어 접근을 시도할 때 마치 실제 하드웨어에 접근하는 것처럼 동작시킴
20. 트랩: 하이퍼바이저가 중간에서 __ OS의 __ 명령들을 가로챌 수 있음.
21. 각 트랩은 __ 스위치를 필요로 하며, 이는 처리해야 할 작업량이 많을 때 성능을 크게 저하시킴.
22. 컨텍스트 스위치 (Context Switch) : 하나의 __가 CPU를 사용 중인 상태에서 다른 프로세스가 CPU를 사용하도록 하기 위해, 이전의 프로세스의 상태를 __하고 새로운 프로세스의 상태를 적재하는 작업
23. Binary Translation: 게스트 OS 내에서 실행되는 __ 전체 코드를 대상으로 __를 분석하고 필요에 따라 특권 명령어를 __ 시스템에서 실행 가능한 명령어로 변환하는 기술, 단점으로는 __ 관리 오버해드가 있다
24. __: 게스트 운영체제(Guest OS)가 하이퍼바이저(Hypervisor)에게 직접 서비스나 작업을 요청함 -> 하이퍼콜, 단점으로는 오픈 소스가 아니면 __에 적용이 쉽지 않음.
25. __: 가상화 환경에서 하이퍼바이저가 제공하는 인터페이스
26. 하드웨어 전가상화 (Hardware Birtualization): 전가상화에서 발생하는 __을 하이퍼바이저가 아닌 CPU에서 바로 처리, CPU가 __와 GUEST OS 간의 전환을 지원하는 __ 지원 기술(VT-x와 AMD-V)을 제공하며 전가상화와 반가상화의 장점을 적절히 혼합( __OS에 제약이 없음(전가상화) + 강력한 성능(반가상화))했다
